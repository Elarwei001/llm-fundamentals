# Day 18: 推测解码(Speculative Decoding)

> **核心问题**:我们怎么才能一次生成多个候选 token,同时又保持与目标模型完全相同的输出分布,并真正让 LLM 推理更快?

---

## 开篇

如果说 KV 缓存(KV Cache)解决的是模型每写一个新词都要把整本书重新翻一遍的问题,那么**推测解码(Speculative Decoding)**解决的,就是让一个更快的小助手先往前草拟几步,再由真正的大模型成批审核。

这个视角很重要,因为现代大语言模型的推理有一个很别扭的地方。目标模型很大,所以每一步解码都很贵。更麻烦的是,自回归生成(Autoregressive Generation)天然是串行的,token 2 依赖 token 1,token 3 依赖 token 2,依此类推。图形处理器(Graphics Processing Unit, GPU)最喜欢的是大规模并行工作,但普通解码给它的却是一个"算一个 token,停一下,再算一个"的小循环。所以模型可能很聪明,但体感仍然会慢。

推测解码就是冲着这个串行瓶颈来的。它不让大模型每次只吐出 1 个 token,而是先引入一个更便宜的**草稿模型(draft model)**。这个小模型先提议一小段候选 token,然后目标模型一次性验证这段候选。如果草稿模型经常猜对,那么目标模型的一次昂贵验证,就可能一下子确认多个 token。换句话说,一次大模型前向不再只推进 1 步,而可能推进 3 步、4 步,甚至更多。

你可以把它想成结对编程。一个初级工程师先把显然的几行代码快速写出来,资深工程师再一次性 review 这一小段。如果前者写得靠谱,项目就会一下子往前跳一截。如果写偏了,资深工程师就在第一个错误的地方纠正,然后从那里接管。速度提升的关键不在于"有没有助手",而在于"助手猜对的概率高不高"。

这篇文章会把它讲清楚,包括机制、数学直觉、接受与拒绝逻辑、系统层面的权衡,以及为什么它很好用,但并不神奇。

---

## 1. 为什么普通解码还有很多性能没挖出来

**一句话总结**:普通自回归推理在每一个 next-token 决策上都依赖最强的目标模型,因此很准确,但也非常串行。

一个 decoder-only 语言模型定义的是

$$
P(x_{1:T}) = \prod_{t=1}^{T} P(x_t \mid x_{1:t-1}).
$$

生成时我们反复做四件事:

1. 用当前前缀运行目标模型,
2. 得到下一个 token 的分布,
3. 采样或选出下一个 token,
4. 把它接回前缀,继续循环。

有了 KV Cache,这个过程已经比"每次都把历史全部重算一遍"高效很多,因为历史 token 的 key 和 value 不需要重复计算。但它的本质仍然没有变,整个过程还是串行的。每想多生成一个 token,就得再咨询一次大模型。

这就意味着,大模型成了整个交互速度的节拍器。如果目标模型每次给出一个 next-token 分布要花 20 毫秒,那么输出 100 个 token,大致就要做 100 次这样的昂贵步骤。推测解码要松动的,就是这个地方。

![图 1:普通解码与推测解码的时间线](./images/day18/speculative-timeline.png)
*图注:普通解码大致是"每生成一个 token,就做一次目标模型前向"。推测解码则希望让便宜的草稿模型先提议多个 token,再用一次目标模型验证来摊薄成本。*

注意,推测解码不是要替代目标模型。目标模型仍然是最终裁判。它要做的,是减少我们"一个 token 一个 token 地等裁判"的次数。

## 2. 核心机制,先起草,再验证

**一句话总结**:一个便宜的小模型先提出一小段延续,再由大模型并行检查这段延续是否可以接受。

假设当前前缀是 $x_{1:t}$。普通解码会直接去问目标模型下一个 token $x_{t+1}$ 是什么。推测解码改成下面这套流程:

1. 用便宜的**草稿模型** $q$ 先生成 *γ* 个候选 token。
2. 把这段草稿接到前缀后面,让**目标模型** $p$ 一次性跑过去。
3. 比较草稿模型的提议与目标模型允许的分布。
4. 接受一段连续的前缀 token。
5. 一旦遇到第一个被拒绝的 token,就从那里回退,并用目标模型修正采样。

为什么这套流程有希望更快?因为目标模型可以在一次前向里对整段草稿做评分和验证。如果前 3 个草稿 token 都能通过,那么一次昂贵的大模型验证,可能就把生成推进了 3 到 4 个位置,而不是只推进 1 个。

> **目标模型怎么"知道"一个 draft token 好不好？**
>
> 目标模型**不是**一个一个检查 token 的。它只需要对整个序列（prefix + 所有 draft tokens）跑**一次前向传播**。Transformer 天然会在一次前向里输出**每个位置**的 token 概率分布。
>
> 所以对于每个 draft token 位置，我们同时得到两个数：
> - q(draft_token)：草稿模型的概率（已经在草稿阶段知道了）
> - p(draft_token)：目标模型的概率（来自这一次前向传播）
>
> 比较很简单：算 p/q，决定接受还是拒绝。不需要额外计算。
>
> **类比：** 老师批改学生预填的答题卡。老师不是一道一道检查，而是把整张答题卡扫一遍，所有答案的概率分布同时出来。然后逐个对比学生预填的答案跟标准答案。
>
> **常见误解：** 目标模型**不是**把 5 个 draft token 拆成 5 次计算再并行。它真的就是**1 次正常的 forward pass**，输入是一整段序列（prefix + draft tokens），只是输入比平时长了几个 token。输出自然就包含了每个位置的概率分布——包括 draft 位置。
>
> **还有一个要点：** 目标模型处理的是整个序列，所以 prefix 部分的计算也没有浪费——它们成了 KV cache 的条目，后续解码步骤可以直接复用。

它背后的直觉并不复杂。如果草稿模型对目标模型来说是一个"还不错的猜题者",那么大量验证步骤都只是在确认它猜得对,而不是推翻它。在这种情况下,原本严格串行的生成过程,就被改造成了"按块推进"的工作流。

但这里有一个非常关键的要求,最终输出必须忠于**目标分布(target distribution)**,而不是草稿分布。否则,我们只是偷偷换成了一个更差的小模型在生成,然后希望运气不错。推测解码真正聪明的地方,是它能通过接受-拒绝逻辑保持精确性。

## 3. 接受与拒绝,为什么最终结果仍然可以完全精确

**一句话总结**:草稿 token 不是无条件接收的,而是通过拒绝采样式的规则决定是否接受,从而确保最终序列的分布和目标模型直接生成时一致。

设 $p(\cdot)$ 是目标模型在某一位置上的分布,$q(\cdot)$ 是草稿模型在同一位置上的分布。当草稿模型提出一个 token $y$ 时,我们**不会**直接收下,而是按下面的概率接受它:

$$
\alpha(y) = \min\left(1, \frac{p(y)}{q(y)}\right).
$$

这个式子很值得品一品:

- 如果草稿模型**低估**了某个 token,相比之下目标模型更喜欢它,那么 $p(y) / q(y) > 1$,接受概率就被截断为 1,直接接受。
- 如果草稿模型**高估**了某个 token,也就是它比目标模型更执着于这个 token,那么接受概率就会低于 1。

简单说:草稿模型保守一点没关系,目标模型几乎都会同意。但如果草稿模型过度自信,提议了目标模型不喜欢的 token,就会被拒绝。

而一旦某个 token 被拒绝,事情也不会简单粗暴地变成"那就重新从目标模型采一个"。真正精确的做法是从一个**修正后的剩余分布(residual distribution)**中采样。概念上它与下面这个量成正比:

$$
\bigl(p(x) - q(x)\bigr)_+,
$$

其中下标 $_+$ 表示"只保留正数部分"——即 $\max(p(x) - q(x),\, 0)$。如果目标模型给某个 token 分配的概率比草稿模型高，差值就保留下来。如果草稿模型高估了，差值为负，直接归零。它表示的是，目标模型原本有的概率质量里，那些还没有被草稿接受过程消耗掉的"剩余额度"。正是这个修正步骤，保证了最终结果在分布上与直接使用目标模型生成完全一致。

> **用大白话解释 "corrected residual distribution":**
>
> 想象在自助餐厅。草稿模型(你朋友)帮你选了 10 道菜。营养师(目标模型)来审核:
> - 8 道菜通过了--这些就是被接受的 draft token。
> - 第 9 道被否了--这就是第一次被拒绝。
>
> 现在营养师要给第 9 道菜换个替代品。但不能随便选,因为前 8 道菜已经"用掉"了一部分理想的概率配额。Residual distribution 就是**扣除已选配额后,剩下的选项**。
>
> 类比成预算:营养师的理想菜单是 p(x)。前 8 道菜已经花掉了一部分预算(通过 q(x))。剩余 (p(x) - q(x))+ 就是剩下的预算--只取大于 0 的部分。替代菜就是从这个剩余里选的。
>
> **为什么不直接从 p(x) 重新采样?** 因为那样会把前 8 道菜已经消耗的概率重复计算。Residual 修正保证最终分布**完全等于**目标模型,不是近似。

你不必死记证明,但最好理解它的味道。想象有两个出纳员在核同一张账单。草稿出纳先把明显的项目填好。目标出纳逐项复核。如果一项在两边都说得过去,就保留;如果草稿那边把某项高估了,目标出纳就把多算的部分扣回去,再从剩余的合法概率里重新抽样。目标不是"差不多就行",而是"在分布意义上完全对齐"。

> **这个规则是谁提出的?**
>
> 接受规则 $\alpha(y) = \min(1, p(y)/q(y))$ 源自经典的**拒绝采样(rejection sampling)**(von Neumann, 1947),一种古老的统计技术。Speculative Decoding 的贡献在于把这个技巧**应用到 LLM 推理加速**上。
>
> 两个团队在 2023 年几乎同时独立提出:
> - **Leviathan, Kalman, Matias (Google)** - *"Fast Inference from Transformers via Speculative Decoding"*
> - **Chen et al. (DeepMind)** - *"Accelerating Large Language Model Decoding with Speculative Sampling"*
>
> **关键性质:** 这个规则保证最终输出分布**完全等于**目标模型 p 的分布--不是近似,而是数学上可证明的等价。这不是牺牲质量换速度,而是**不损失任何质量的加速**。
>
> **类比:** 想象在自助餐厅,朋友(草稿模型)帮你先选了菜,营养师(目标模型)来审核。如果营养师觉得没问题就直接吃,如果觉得不太好就有一定概率让你换一道。最终你吃到的菜的营养分布,跟营养师亲自给你选的**一模一样**。

![图 2:接受前缀与第一次拒绝](./images/day18/acceptance-walk.png)
*图注:验证过程通常会先接受一段连续的草稿前缀,然后在第一个不合适的位置停下来。这种"接受前缀"的结构,正是推测解码产生加速的主要来源。*

很多简化版介绍会把它讲成"只要目标模型同意就继续,不同意就拒绝"。这个直觉作为入门是有帮助的,但真正的算法比这更精细,因为它要保住的不是 greedy 输出,而是目标模型完整的采样分布。

## 4. 为什么它真的可能更快

**一句话总结**:推测解码的速度提升,来自用更少的大模型验证步数,换掉原本大量逐 token 的昂贵决策。

一个足够实用的心智模型是:

- 普通解码,大约是每生成 1 个 token 付出 1 次目标模型前向;
- 推测解码,则是每个候选 token 付出廉价草稿成本,再对一整块候选付出 1 次目标模型验证成本。

如果草稿块长度是 $\gamma$,而且被接受的前缀长度通常比较长,那么每做一次目标模型前向,平均就能推进多个 token。这就是它的核心收益来源。

一个常见的工程近似式是:

$$
\text{speedup} \approx \frac{1 + \gamma a}{1 + c\gamma},
$$

其中:

- $a$ 是接受率(acceptance rate),
- $\gamma$ 是草稿长度,
- $c$ 是草稿模型相对于一次目标模型验证的成本比例。

这不是论文中的严格主定理,但它非常适合建立工程直觉,因为它一下子就告诉你三件事:

1. 接受率越高越好;
2. 草稿块越长,不一定越好,前提是接受率不能明显变差;
3. 草稿模型如果本身太贵,会把收益吃掉。

![图 3:速度提升高度依赖接受率](./images/day18/speedup-vs-acceptance.png)
*图注:不同草稿长度下的近似加速曲线。若草稿模型经常猜错,额外的草拟工作就不值得;若它经常猜对,加速就会变得相当可观。*

所以推测解码并不是免费午餐。它最适合的情况,是草稿模型足够便宜,而且又足够像目标模型,能让长前缀经常通过验证。

## 5. 系统视角,为什么 GPU 喜欢这种工作形状

**一句话总结**:推测解码把目标模型的工作从"频繁、细碎、串行"改造成"更少次、但更成块"的验证任务,更符合现代加速器的习性。

GPU 喜欢的是宽并行、大批量 kernel,而不是一个一个很窄的小循环。普通解码即使有 KV Cache,本质上还是在不停地问,"再给我一个 token","再给我一个 token"。目标模型反复被调用去做一个非常窄的任务。

推测解码改变的是**工作负载的形状**。草稿模型先快速跑一小段,目标模型再一次性验证多个位置。这带来两个系统层面的好处:

1. 减少了与目标模型频繁同步的次数;
2. 让目标模型每次出手时,都能处理更密集、更有并行度的验证工作。

> **这四个概念大白话解释：**
>
> **1. "宽并行 kernel"：** GPU 就像 8 车道高速公路,设计给大量车辆并行开的。普通解码每次只走 1 辆车（每步 1 个 token）,大部分车道闲置。推测解码一次处理一整块 draft,填满更多车道。
>
> **2. "减少同步点"：** 普通解码：算 1 个 token,同步,读结果,决定下一个,再算,再同步... 推测解码：一次性生成一整块 draft,一次性验证。停顿次数少了,开销就小了。类比：送快递,一趟送 10 个包裹 vs 跑 10 趟每个送 1 个。
>
> **3. "更密集的验证工作"：** 每次调用目标模型,现在验证多个 token 而不是 1 个。同样的 GPU 启动开销,干了更多活。出租车载 1 人 vs 公交车载 30 人——同一趟,吞吐量天壤之别。
>
> **4. "分支预测类比"：** CPU 有个经典技巧：遇到 if/else,先猜一条路往下执行。猜对了就赚了,猜错了回滚重来。推测解码完全一样的思路——草稿模型猜,目标模型验证,猜错了回滚修正。

这也是为什么推测解码的收益很依赖你的 serving stack。如果你的目标模型已经被优化得极其极致,那么额外收益可能有限;如果你的系统本来就被 decode 阶段的小步串行拖住,那么推测解码往往会更有价值。现实世界通常落在这两个极端之间。

还有一个挺好用的类比,它有点像 CPU 的分支预测(branch prediction)。处理器先猜接下来会走哪条路径,提前执行,如果猜错就回滚。这个类比不是完全一致,但足够帮助理解:预测很便宜,回滚是可以接受的,只要预测准确率够高,总吞吐量就会提升。

## 6. 草稿模型怎么选,这是最核心的权衡

**一句话总结**:最好的草稿模型,不是最聪明的那个,而是"便宜到足以前冲,同时又靠谱到经常被接受"的那个。

这是很多第一遍解释会略过的地方。直觉上你可能会觉得,草稿模型越强越好。其实未必。

更大的草稿模型,通常会让接受率更高,因为它的分布更接近目标模型。但它本身也更贵。一个极小的草稿模型虽然便宜,却可能把续写猜得乱七八糟,导致目标模型在很早的地方就拒绝它。两边都可能输。

![图 4:草稿质量与草稿成本的平衡](./images/day18/draft-model-tradeoff-map.png)
*图注:一个真正有用的草稿模型,通常处在中间区域,既便宜到可以先跑几步,又准确到足以让目标模型频繁接受它的提议。*

实际部署时,最佳点取决于很多因素,比如模型家族是否一致、tokenizer 是否兼容、硬件是什么,以及**业务场景(workload)**。举个例子,聊天场景中结构化、重复性强的请求,接受率可能更高;而开放式创作则更难预测。

所以,推测解码常常更像一个系统优化问题,而不是一个单纯的采样技巧。理论算法本身很优雅,但要在生产环境里真的跑出收益,关键还是 draft-target 这对搭档选得对不对。

## 7. 变体,从经典双模型到 self-speculative 方法

**一句话总结**:后续很多工作都在保留核心思想的同时,尽量减少单独维护一个草稿模型的成本。

### 7.1 经典推测解码

这是 Leviathan 等人在 2023 年提出的标准形态。一个较小的外部模型起草,一个较大的目标模型验证,最终输出分布与大模型直接生成完全一致。

### 7.2 Speculative Sampling 及相关表述

有些论文会使用 slightly different 的表述、接受规则或性能分析方式,但本质目标很一致,都是想把多个候选 token 打包验证,同时不牺牲目标模型级别的采样质量。

### 7.3 Medusa 与多头草拟

像 **Medusa** 这样的工作,不再依赖一个完全独立的外部草稿模型,而是在主模型上外挂多个解码头,直接并行预测更远位置的 token,再统一验证。这样做的好处是部署可能更简单,因为你不一定要额外托管第二个模型,但通常也需要额外训练或结构修改。

### 7.4 Self-speculative decoding

还有一些方向会尝试让同一个大模型的"便宜路径"先给出粗略草稿,再由完整路径确认。例如利用早退层(early exit)或者更轻量的内部计算路径。这个思路很诱人,因为它想实现"同一个模型,自己先猜,再自己审"。

### 7.5 草稿模型怎么训练?

获得一个好的草稿模型对接受率至关重要。不同的训练策略各有取舍：

**方法 1：直接用现成的小模型（无需训练）**
- 直接把同家族的小模型（如 LLaMA-68M）和大模型（如 LLaMA-70B）配对使用。
- 优点：零训练成本，部署简单。
- 缺点：如果小模型在你的具体领域跟目标模型不对齐，接受率可能很低。
- 最适合：两个模型共享 tokenizer 和架构。

**方法 2：从目标模型做知识蒸馏**
- 训练草稿模型去模仿目标模型的输出分布。目标模型当"老师",草稿模型当"学生"。
- **离线蒸馏**：先收集目标模型在数据集上的输出，然后用这些 (输入, 目标输出) 对训练草稿模型。研究表明离线蒸馏比在线蒸馏效果好 11-25%（Hong et al., 2025）。
- **在线蒸馏**：草稿模型实时生成 token，目标模型在训练过程中提供反馈。
- **DistillSpec**（Zhou et al., 2024）：使用草稿模型自己生成的 on-policy 数据和任务特定的散度函数。关键发现：用草稿模型*实际*产出的数据训练，比用静态的老师数据更有效。
- **EAGLE**：不直接预测 token，而是训练草稿模型预测目标模型的*隐藏特征*（LM head 之前那一层），然后用目标模型的 LM head 把特征转成 token。这样直接利用了目标模型的表示空间。

**方法 3：N-gram 和检索式方法（无需模型训练）**
- **N-gram 草稿**：用统计 N-gram 模型根据当前上下文的频率模式预测下一个 token。极其便宜，但只能捕捉浅层模式。
- **检索增强草稿**：在数据库中查找类似上下文，用那些补全作为草稿候选。适合代码补全和模式重复的领域任务。
- 优点：不需要运行神经网络做草稿，内存开销最小。
- 缺点：仅限于重复性高或结构化的文本。

**方法 4：领域适配**
- 从通用草稿模型开始，在领域特定数据上微调（如生物、代码、中文）。
- 研究表明，用目标模型生成的合成数据（通过 Magpie 等框架）可以达到真实用户查询训练效果的 80-93%。
- 适合特定部署场景（如编程助手、医疗聊天机器人）。

> **该选哪种方法？**
>
> - **快速上手**：用同家族现成小模型（方法 1）
> - **最高质量**：知识蒸馏（方法 2），尤其是离线蒸馏
> - **最低开销**：N-gram 或检索式（方法 3），适合代码/结构化文本
> - **特定领域**：在领域数据上微调通用草稿模型（方法 4）

这些变体追求的是同一个梦想,减少串行依赖,但不放弃目标模型级别的质量。

## 8. 局限与失效模式

**一句话总结**:只有当草稿模型与目标模型合作得足够好时,推测解码才真正有收益,否则它会额外增加工作和复杂度。

下面是几个主要失效模式。

### 8.1 接受率太低

如果草稿模型经常猜错,目标模型会很早拒绝,那么加速就很有限。你已经为草稿模型付了成本,但没有换来足够长的 accepted prefix,最后甚至可能比普通解码更慢。

### 8.2 草稿模型本身的额外负担

多跑一个模型,意味着更多显存、更多调度逻辑、更多工程维护,有时还会牵扯 tokenizer 兼容、KV cache 协调、批处理策略等问题。算法可能很漂亮,但生产系统会变复杂。

### 8.3 验证依然离不开大模型

推测解码减少的是"大模型出手的频率",不是把大模型彻底拿掉。如果你的瓶颈几乎全部来自目标模型验证,那么可提升的上限仍然有限。

### 8.4 对 workload 很敏感

接受率会随任务变化。结构化摘要、模板化回复可能更容易草拟;开放式创作更难;代码补全则经常是"有些片段很容易猜,有些片段极其严格"。

### 8.5 与批处理和服务系统的耦合

在多请求并发场景下,要让两套模型都被高效利用,并不简单。论文里的单请求结果很好看,不代表在生产集群里就能等比例兑现。

## 9. 代码示例,一个玩具版推测解码循环

**一句话总结**:实现上比普通解码多了不少 bookkeeping,因为你必须正确处理 accepted prefix 和 rejection 之后的修正采样。

```python
import random
from typing import List


def sample_from(dist):
    # dist 是一个 dict: token -> probability
    r = random.random()
    cumulative = 0.0
    for token, prob in dist.items():
        cumulative += prob
        if r <= cumulative:
            return token
    return list(dist.keys())[-1]


def accept_prob(target_dist, draft_token, draft_dist):
    p = target_dist.get(draft_token, 0.0)
    q = draft_dist.get(draft_token, 1e-12)
    return min(1.0, p / q)


def residual_distribution(target_dist, draft_dist):
    # 目标分布中尚未被草稿过程"覆盖"的剩余概率质量
    residual = {}
    total = 0.0
    for tok, p in target_dist.items():
        value = max(0.0, p - draft_dist.get(tok, 0.0))
        residual[tok] = value
        total += value
    if total == 0.0:
        return target_dist
    return {tok: value / total for tok, value in residual.items()}


def speculative_step(prefix: List[str], draft_model, target_model, gamma: int = 4):
    # 1) 草稿模型自回归地提出多个候选 token
    draft_tokens = []
    draft_dists = []
    ctx = prefix[:]
    for _ in range(gamma):
        q = draft_model.next_token_distribution(ctx)
        y = sample_from(q)
        draft_tokens.append(y)
        draft_dists.append(q)
        ctx.append(y)

    # 2) 目标模型一次性对整段草稿做并行评分
    target_dists = target_model.block_distributions(prefix, draft_tokens)

    # 3) 持续接受,直到遇到第一个拒绝
    accepted = []
    for y, q, p in zip(draft_tokens, draft_dists, target_dists):
        if random.random() <= accept_prob(p, y, q):
            accepted.append(y)
        else:
            corrected = residual_distribution(p, q)
            accepted.append(sample_from(corrected))
            return prefix + accepted

    # 4) 如果整段草稿都通过,再由目标模型补采一个新 token
    accepted.append(sample_from(target_model.next_token_distribution(prefix + accepted)))
    return prefix + accepted
```

这个玩具版本省略了 KV Cache、批处理、张量并行和工程级高效实现,但它保留了最关键的逻辑结构:先起草,再验证,接受一段前缀,遇到第一次拒绝后从修正分布回退采样。

## 10. 常见误解

### ❌ "推测解码会改变模型行为。"

在经典的精确算法里,不会。最终输出分布与目标模型直接生成一致。它追求的是加速,不是近似替代。

### ❌ "草稿 token 越多,加速一定越好。"

不对。只有当草稿模型仍然足够准确时,更长的草稿块才有意义。否则拒绝太早,收益会被吃掉。

### ❌ "它能替代 KV Cache。"

不能。两者解决的是不同问题。KV Cache 避免重复计算历史 key/value,推测解码则是减少目标模型必须逐 token 给出 next-token 分布的次数。

### ❌ "接受率只是个小调参项。"

恰恰相反。接受率是整个系统最核心的经济变量之一。一个接受率很差的推测解码系统,完全可能比普通解码更慢。

## 总结

| 概念 | 一句话解释 |
|------|------------|
| 推测解码 | 用便宜的草稿模型先提出多个 token,再由目标模型成批验证。 |
| 接受规则 | 通过拒绝采样式的接受逻辑,保证最终分布仍然与目标模型一致。 |
| 加速来源 | 一次昂贵的目标模型验证,可能推进多个已接受 token。 |
| 主要瓶颈 | 真正收益取决于接受率和草稿模型成本。 |
| 核心权衡 | 理想的草稿模型必须既便宜,又足够靠谱。 |

**关键 takeaway**:推测解码是现代 LLM 推理里一个很漂亮的想法,因为它真正瞄准了敌人,串行依赖。它不会让目标模型消失,也不会让长上下文生成变成免费的。但它做了一件更实际、也更有意思的事,把生成过程改造成"快的猜题者 + 慢的裁判"协作模式。当这对搭档选得好时,体感速度会明显变快,而输出分布仍然保持目标模型级别的严谨性。
