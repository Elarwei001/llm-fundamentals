# Day 23: 数学与逻辑

> **核心问题**：为什么语言模型明明“看起来很聪明”，却还是会在数学和逻辑上犯很蠢的错误？真正有效的补救方法又是什么？

---

## 开篇

LLM 最奇怪的一点之一是，它可以解释贝叶斯公式、总结论文、写出很漂亮的 Python，然后下一秒在一个小学应用题上翻车，只因为中间少了一个负号。

这件事之所以让人不舒服，是因为它很违和。一个模型既然能讨论抽象代数，为什么却不能稳定数清 *strawberry* 里有几个 `r`？既然能解释动态规划，为什么又会在一个简单排班谜题上突然崩掉？

短答案是：**语言流畅**和**符号推理可靠**不是同一种能力。LLM 的训练目标，是去延续“看起来合理”的 token 序列。可数学和逻辑要求的东西更苛刻，它们要求精确维护中间状态、精确做变换、精确做验证。自然语言是宽容的，符号问题不是。写文章时，稍微模糊一点，读者通常还能接受。做代数时，一个中间 token 错了，最后答案就可能全毁。

你可以把它想成“做一场有说服力的演讲”和“对一张表格做结算”的区别。两者都需要智能，但后者对细小错误的惩罚大得多。LLM 天然更擅长前一种任务，对后一种任务则只有部分擅长。

所以，现代 LLM 推理的真正故事并不只是“把模型再做大一点”。更关键的是：什么时候语言建模本身已经够用，什么时候不够；以及为什么代码执行器、计算器、定理证明器、验证器这些工具，常常比又一个花哨 prompt 更重要。

这篇文章会拆开这个问题。我们会讨论为什么数学和逻辑是特殊压力测试，LLM 到底会在哪里失手，benchmark 提升说明了什么、又没有说明什么，以及工程上怎样把一个脆弱的语言模型，变成更可靠的推理系统。

## 1. 为什么数学和逻辑是特殊的压力测试

**一句话总结**：数学和逻辑会暴露自然语言任务里容易被掩盖的问题，因为它们要求的是精确中间状态，而不是“看起来合理”的延续。

语言模型的训练目标本质上是预测下一个 token：

$$
\mathcal{L} = -\sum_{t=1}^{T} \log P(x_t \mid x_{1:t-1}).
$$

这个目标非常强大。它能学会语法、事实、风格、代码习惯，甚至不少潜在算法。但训练目标本身并没有显式写着“维护一个形式上有效的证明状态”，也没有写着“绝不能丢一个进位”。模型学到的，是那些最能降低预测损失的内部机制。

之所以这已经足够支撑很多任务，是因为语言数据里本来就包含大量弱监督的推理痕迹。教材里有推导，代码仓库里有调试过程，数学论坛里有例题解法。所以模型会学到，当一个问题长得像 A 类时，后面往往跟着某种 B 类步骤。

但数学和逻辑比开放式文本生成更不宽容，至少有三个原因。

### 1.1 局部合理，不等于整体正确

一句话可以局部看起来很顺，哪怕略微含糊，读者也常常会原谅。但一个证明步骤或者算术步骤不能只是“差不多对”。乘法里进位错一次，最后答案就是错的。

![图 1：多步数学里的误差放大](./images/day23/math-error-amplification.png)
*图注：随着依赖性操作的数量增加，越早出现的小错误，越容易级联到最终答案。接入工具后，曲线会平得多。*

### 1.2 中间状态比措辞更关键

很多逻辑题真正依赖的是隐式记账：哪个假设还有效，哪个约束还在起作用，哪个对象对应哪个实体。人类做这类题时，通常会把这些状态写在纸上。纯 LLM 则要把它们压进激活状态和生成文本里，这当然可能做到，但非常脆。

### 1.3 符号任务天然“克制”近似计算

神经网络本质上是非常强的近似学习器，而符号推理经常是一个精确游戏。你要么保住了变量绑定，要么没有；你要么尊重了量词约束，要么没有。近似计算和精确符号操作之间的这道缝，正是 LLM 推理里的核心张力之一。

这也是为什么数学和逻辑会成为这么有代表性的测试。它们并不是智能的全部，但它们特别擅长暴露一个系统到底有没有在全过程里把结构维护好。

## 2. 为什么 LLM 看起来比实际更会推理

**一句话总结**：高 benchmark 分数确实代表进步，但也可能掩盖脆弱性，因为很多题可以靠模板识别加一点 scratchpad 推理撑过去。

如果说 LLM 完全不会数学和逻辑，那显然不对。和早期模型相比，它们已经强了很多。Chain-of-Thought（CoT，思维链）、self-consistency（自一致采样）、程序生成、推理时搜索，都让 GSM8K、MATH、代码题等任务出现了明显提升。

但如果把这些提升直接理解为“基础模型已经变成稳健符号推理器”，那也不对。

微妙的地方在于，benchmark 成功可能是几种能力混在一起的结果：

1. **模式检索（pattern retrieval）**：识别出这题像训练中见过的大量例子。
2. **部分抽象（partial abstraction）**：学会一些可复用模板，比如“设变量，列方程，求解”。
3. **草稿执行（scratchpad execution）**：利用额外 token 暂存中间状态。
4. **答案筛选（answer selection）**：采样多条推理路径，再选最常见结果。

这四种能力都很有用，但只有其中一部分，真的对应我们想要的那种“在分布变化下也稳定”的系统化推理。

最近关于 symbolic perturbation（符号扰动）的工作就很能说明问题。模型可能能做对一个熟悉模板的应用题，但当你把同一层数学结构换一个名字、换一下数字出现顺序，或者插入一点无关细节后，它就开始掉点。表面故事变了，抽象问题却没变。如果准确率明显下滑，通常说明模型更多依赖的是熟悉模式，而不是不变的符号结构。

![图 2：Benchmark 表现可能高估鲁棒性](./images/day23/benchmark-vs-robustness.png)
*图注：Benchmark 上的提升是真实的，但一旦把同一个符号问题改写或扰动，鲁棒性往往掉得更快。*

这有点像一个学生，平时作业做得很好，因为他认得章节套路；但考试一旦把同一个概念换个问法，他就开始不稳了。你不能说他什么都没学会，但也不能说他已经掌握了可迁移的抽象能力。

所以，当人们问“LLM 现在会做数学了吗”，最诚实的回答其实是：**在 benchmark 上，经常会；在你真正关心的抽象意义上，不总是。**

## 3. 数学与逻辑中的核心失败模式

**一句话总结**：最常见的问题是算术漂移、变量绑定脆弱、过早套模板，以及验证太弱。

我们说得更具体一点。LLM 到底是怎么错的？

### 3.1 算术漂移

LLM 可以“模拟”算术过程，但模拟不等于精确执行。多位数运算、模运算、连续计算，在纯自然语言环境下依然很脆。

一个很有用的心智模型是：模型常常知道**应该使用什么算法**，但不一定能把这个算法执行到最后一位都不出错。这也是为什么让模型去写 Python 或调用计算器，通常能明显提升结果。难点很多时候不在“理解题目”，而在“把精确计算做对”。

### 3.2 变量绑定和指代混乱

逻辑题常常依赖于谁做了什么、哪个集合包含什么、哪个条件对应哪个对象。一个代词滑掉、一个变量对错了，后面的推理链就会悄悄坏掉。

这类问题在那些看起来像“文字小游戏”的谜题里尤其常见，比如座位安排题、真假话题。模型可能整段话都写得很顺，但内部绑定早就丢了。

### 3.3 过早的模板匹配

LLM 的解码目标是产出高概率 continuation。面对难题时，这会带来一种常见偏差：它太早认出一个“看起来像”的模式，然后在还没确认细节是否真的匹配前，就跳进一个熟悉的解法模板。

这也是为什么“think step by step”有时有效。它的作用之一，就是让这个“过早跳跃”慢下来。它不保证正确，但至少给了模型更多机会在落地前发现矛盾。

### 3.4 自我检查不够强

很多人以为，既然模型能作答，那让它“再检查一遍”就好了。可生成答案和验证答案并不是对称任务。如果第一次犯错的偏差，在第二次检查时还在，那么模型很可能只是换一种更自信的口气，把错误重新确认一遍。

一个 verifier 真正有帮助，往往是因为它拥有不同的归纳偏置、不同的证据源，或者直接能访问精确工具。

![图 4：神经系统和符号系统的强项并不一样](./images/day23/neural-vs-symbolic-strengths.png)
*图注：纯 LLM 在语义模式识别上很强，但在精确算术、形式约束处理和验证上通常不如符号工具。真正可靠的系统会把两者结合起来。*

### 3.5 全局一致性丢失

在长证明或长规划任务里，每一个局部步骤都可能看起来合理，但整个链条还是会慢慢偏离最初约束。这也是为什么长程逻辑仍然难。模型很擅长生成“附近看起来通顺”的文本，但要跨很多步保持全局符号一致性，要求高得多。

## 4. 为什么工具能帮这么多

**一句话总结**：工具之所以有效，是因为它让 LLM 做自己擅长的事，也就是理解语言和拆解问题，同时把精确执行交给真正擅长精确的系统。

这是对工程实践最重要的一段。

如果你问，一个纯 LLM 是否应该在内部独自完成所有符号计算，我的答案通常是否定的。这不是哲学上的贬低，而是朴素的工程常识。

计算器更适合做算术。SAT solver（可满足性求解器）更适合处理形式约束。SymPy 更适合代数变形。编译器和测试集更适合验证代码。

![图 3：混合式推理流水线](./images/day23/reasoning-tool-pipeline.png)
*图注：在更可靠的推理系统里，LLM 负责理解和拆解问题，工具负责精确计算与检查。*

现代系统的典型模式因此变成：

$$
\text{可靠推理系统} \approx \text{LLM} + \text{scratchpad} + \text{工具执行} + \text{验证}.
$$

这个拆分很关键，因为它把两类完全不同的问题分开了。

- **语义理解（semantic interpretation）**：题目到底在问什么？哪些变量重要？该用什么公式或算法？
- **精确执行（exact execution）**：一旦结构确定，精确答案到底是什么？

纯文字版 Chain-of-Thought 其实是在要求 LLM 用同一种介质同时完成这两件事。而 PAL（Program-Aided Language Models）和 PoT（Program of Thoughts）这类方法，会把这两件事拆得更干净。模型先把题目翻译成代码，再由 Python 或符号库去做真正的精确计算。

这个设计有时甚至有效得有点“朴素”。就像你有一个非常聪明、分析能力极强、但心算偶尔会出错的实习生。你不会硬逼他把所有数字都在脑子里算完，你会直接给他 Excel。

## 5. 一个小代码例子：把代码当成精确计算器

**一句话总结**：很多所谓“推理”问题，真正的胜负手不是写更长的 prose，而是把问题转成可执行结构。

下面是一个很小的 SymPy 例子。LLM 的工作，是把题目正确翻译成方程；求解器的工作，是精确求解。

```python
# 用符号数学求解一个简单应用题。
import sympy as sp

# 题目：
# “Alice 买了 3 本笔记本和 2 支笔，共 19 美元。
#   Bob 买了 2 本笔记本和 5 支笔，共 21 美元。
#   问一支笔和一本笔记本各多少钱？”
notebook, pen = sp.symbols('notebook pen', real=True)

sol = sp.solve(
    [
        sp.Eq(3 * notebook + 2 * pen, 19),
        sp.Eq(2 * notebook + 5 * pen, 21),
    ],
    [notebook, pen],
)

print(sol)
# {notebook: 53/11, pen: 25/11}
```

在这个流程里，LLM 应该贡献什么？

1. 识别出这里有两个未知数。
2. 把文字翻译成方程。
3. 判断这里用符号求解，比纯 prose 算代数更稳。
4. 再把结果翻译回自然语言讲给用户听。

如果你在乎准确率，那 LLM 最不该坚持自己在 prose 里完成的部分是什么？就是最后那段精确代数消元本身。

这其实是一个很重要的观念变化。我们不需要模型自己成为计算器，我们需要它知道**什么时候该用计算器**。

## 6. 数学推导 [Optional]

这一节给想看得更形式化一点的读者。

假设一个推理任务在给出最终答案 $y$ 之前，需要经过一串中间状态 $s_1, s_2, \ldots, s_n$。一个直接作答模型试图学习的是：

$$
P(y \mid x).
$$

而 Chain-of-Thought 模型会把一个文本化轨迹 $r = (r_1, \ldots, r_m)$ 显式写出来，相当于把过程拆成：

$$
\begin{aligned}
P(y \mid x) &= \sum_r P(y, r \mid x) \\
             &= \sum_r P(y \mid r, x) P(r \mid x).
\end{aligned}
$$

self-consistency 的做法，就是对多个 $r$ 进行采样，再对最终答案做聚合，从而近似这个边缘化。

但数学正确性常常近似呈现“按步相乘”的特征。如果每一步中间变换正确的概率是 $p$，而整个解答需要 $n$ 个依赖性精确步骤，那么一个粗略近似是：

$$
P(\text{所有步骤都正确}) \approx p^n.
$$

即便 $p$ 已经很高，比如 $p = 0.95$，随着步骤变长，整条链的可靠性仍然会明显下降：

$$
0.95^{10} \approx 0.60.
$$

这其实就是为什么工具这么有用的更深层原因。外部计算会把那些“必须精确”的子步骤的有效正确率拉高，于是端到端失败概率增长得就慢得多。

## 7. 常见误解

**一句话总结**：最大的误解，一类是把数学失败当成模型“根本没智能”，另一类是把 benchmark 胜利当成问题已经彻底解决。

### ❌ “如果模型连算术都会错，那它就没有推理能力。”

这说得太重了。一个模型完全可能已经具备真实的问题分解、抽象、规划能力，但仍然不擅长精确数值执行。

### ❌ “有了 Chain-of-Thought，数学推理就解决了。”

它确实有帮助，有时帮助还很大，但它不是魔法。它提供了更多工作空间，也可能让错误更容易暴露，但不保证推理过程忠实，更不保证每一步都精确。

### ❌ “模型做得越来越大，工具迟早就不需要了。”

对于某些窄任务也许会，但在实际系统里，工具仍然很有吸引力，因为它们便宜、精确、可审计、而且模块化。

### ❌ “Benchmark 已经能告诉我们模型到底懂不懂数学。”

Benchmark 当然有用，但远远不够。改写后的稳定性、符号扰动下的表现、对抗式变体下的表现，同样重要。

## 8. 给构建者的实践建议

**一句话总结**：把数学和逻辑看成“路由问题”，不要只把它看成“prompt 问题”。

如果你正在构建 LLM 系统，一个更务实的操作手册是：

1. **先识别什么时候精确性真的重要。** 税务、金融、排程、合同、证明、生产代码，这些都值得更严格的推理流水线。
2. **先拆解，再执行。** 让模型把自然语言映射成结构，再调用工具。
3. **优先使用可验证表示。** 代码、方程、表格、JSON 约束、测试，比 prose 更容易检查。
4. **尽量做独立验证。** 换一个模型、换一个符号求解器、或者直接跑可执行测试，都比让同一个模型“再看一眼”更可靠。
5. **测鲁棒性，不只测平均分。** 改写题目、打乱无关细节、换名字换顺序，看看答案是否还能稳定。

更深一点的教训是，数学和逻辑并不是 LLM 故事里尴尬的例外，它们恰恰是最能说明问题的样本。它们告诉我们，部署级智能往往是**组合式的（compositional）**。语言模型提供灵活理解，外部工具提供精确执行，验证机制提供可信度。

我很怀疑这只是一个临时补丁。更可能的情况是，这本来就是可靠 AI 系统的长期形态。

## 9. 延伸阅读

### Beginner
1. [Chain-of-Thought Prompting Elicits Reasoning in Large Language Models](https://arxiv.org/abs/2201.11903)  
   让“let's think step by step”真正变成研究方向的经典论文。

2. [Self-Consistency Improves Chain of Thought Reasoning in Language Models](https://arxiv.org/abs/2203.11171)  
   解释为什么采样多条推理路径，常常比只走一条贪心链更好。

### Advanced
1. [Program-Aided Language Models (PAL)](https://arxiv.org/abs/2211.10435)  
   很清楚地展示了如何把自然语言理解和代码执行拆开。

2. [Program of Thoughts Prompting](https://arxiv.org/abs/2211.12588)  
   强调数值推理在外部化计算后会更可靠。

3. [GSM-Symbolic: Understanding the Limitations of Mathematical Reasoning in Large Language Models](https://arxiv.org/abs/2410.05229)  
   很值得看的一篇提醒，说明 benchmark 提升并不自动等于稳健符号推理。

### Papers
1. [Solving Quantitative Reasoning Problems with Language Models](https://arxiv.org/abs/2206.14858)
2. [Training Verifiers to Solve Math Word Problems](https://arxiv.org/abs/2110.14168)

## 总结

| 概念 | 一句话解释 |
|------|------------|
| Arithmetic drift | 小的中间计算错误会在符号任务中快速累积。 |
| Chain-of-Thought | 额外的推理 token 提供了工作空间，但不保证正确。 |
| Tool use | 外部计算器、代码和求解器能提供精确执行。 |
| Verification | 用独立机制检查答案，能显著提升可靠性。 |
| Robustness | Benchmark 高分不等于在改写或扰动下仍然稳定。 |

**Key Takeaway**：LLM 在数学和逻辑上并不是毫无能力，但它也不是天然值得信任的符号机器。它真正擅长的，往往是理解问题、拆解任务、再把答案讲清楚。只要任务要求精确，最佳设计通常就是混合式系统：让 LLM 负责理解，让工具负责执行，让验证器决定这个答案是否真的值得信任。
