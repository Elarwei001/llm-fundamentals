# Day 22: 推理能力

> **核心问题**：LLM 真的会推理吗，还是只是非常擅长模仿“推理的表面形式”？

---

## 开篇

这是现代 AI 里最容易引发争论的问题之一：*语言模型到底会不会推理*？

一派认为会。毕竟，强一些的模型已经能解奥数风格的数学题、完成多步代码修改，还能用很清楚的自然语言解释自己的思路。另一派认为不会。他们的理由是，模型本质上只是做 next-token prediction（下一个 token 预测），看起来像推理的东西，很多时候只是把训练中见过的大量模式重新组合了一遍。

这两派其实都抓住了一部分事实。

如果你让模型翻译一句话、总结一段文字，或者延续一个熟悉的模式，它往往几乎不用“停顿”就能给出答案。但如果你让它解一个有干扰项的文字题、一个需要跟踪约束的逻辑题，或者一个“前面错一步，后面全崩”的规划任务，表现就常常会随着 scratchpad（草稿空间）、verifier（验证器）、额外搜索、或者工具调用而显著变化。这说明了一件很重要的事：“推理”不是一个神秘的单一开关，而是一组能力的组合，包括问题分解、维护中间状态、搜索候选路径、检查一致性，以及在验证后选择更可靠的答案。

可以把它类比成心算。人类回答 $7 + 5$ 基本靠直觉记忆，但计算 $847 \times 96$ 时，很多人就需要草稿纸。我们不会说前者只是“模式匹配”，后者才算“真正推理”。更合理的说法是，同一个大脑会根据任务难度和可用外部辅助，切换不同工作模式。LLM 也越来越像这样。有些题预训练先验就够了，有些题则非常依赖显式中间计算。

所以真正好的问题不是“推理还是模式匹配”，而是：**从 next-token prediction 里，究竟能涌现出哪些推理能力，这些能力会在哪些地方失效，又该靠什么系统设计把它们变得更可靠？**

这篇文章会把“表面流畅”与“真实解题能力”区分开来，解释为什么 Chain-of-Thought（CoT，思维链）有效，讨论“语言化推理过程是否可信”的争议，并把这些内容和 test-time compute（测试时计算）、工具使用、当前局限联系起来。

---

## 1. 人们说的“推理”到底指什么

**一句话总结**：在 LLM 语境里，推理通常不是指“会说得像在思考”，而是指在需要中间状态、约束跟踪或搜索的任务上，能稳定地产生正确答案。

“推理”这个词本来就很模糊。日常语境里，它可能只是“认真思考”。在认知科学里，它可能指规则遵循、因果推断、抽象能力或规划能力。在 LLM 评测里，它通常更具体一些：模型能不能解决那些**不能只靠局部词汇联想就轻松完成**的任务？

一个比较实用的操作性定义是：

> 如果一个任务要成功，必须把多块信息通过若干中间步骤组合起来，而模型不仅给出答案，还能稳定完成这些中间转换，那么我们就说它表现出了一定的推理能力。

这个定义仍然有灰区，但足够帮助我们粗分任务类型。

1. **识别型任务（recognition-heavy）**：翻译、改写、风格转换、基础事实回忆。这些更多依赖预训练学到的模式。
2. **计算型任务（computation-heavy）**：算术、符号变换、带约束的规划。这些通常需要中间状态，不能全靠“一眼看出”。
3. **搜索型任务（search-heavy）**：定理证明、需要回溯的程序合成、谜题求解。这类任务往往受益于分支搜索、验证和外部工具。

一个很关键的细节是，所有这些任务里都存在某种 pattern matching（模式匹配）。即便是人类推理，也离不开长期学习得到的先验。真正的问题不是“有没有模式匹配”，而是：**对于这个任务，学到的模式本身够不够，还是必须依赖显式机制来维护和检查多步结构。**

![图 1：推理不是非黑即白](images/day22/reasoning-task-map.png)
*图注：有些任务主要由潜在先验解决，而另一些任务越来越依赖显式的中间计算或搜索。*

这也是为什么相关争论经常鸡同鸭讲。当一个研究者说“LLM 会推理”时，他可能是在说“LLM 已经能解决过去被认为需要推理的很多任务”。而另一个研究者说“LLM 不会推理”时，他可能是在说“它的内部过程仍然脆弱、不忠实，也不像系统化的符号推理”。这两句话都可能部分正确。

## 2. 为什么 next-token prediction 会长出“像推理”的东西

**一句话总结**：大规模的下一个 token 预测，可能隐式学到很多“潜在算法”，因为语言和代码里本来就携带着大量人类推理过程的痕迹。

乍一看，next-token prediction 似乎太弱了，怎么可能支持真正推理？训练目标只是

$$
\mathcal{L} = -\sum_{t=1}^{T} \log P(x_t \mid x_{1:t-1}).
$$

这里没有一个显式叫“belief（信念）”的变量，也没有内置 theorem prover（定理证明器），更没有任何硬保证说模型一定学到了正确的世界模型。那为什么推理味道这么重的能力会出现？

因为训练信号远比表面上丰富。

人类写下的文本里，不只有结论，还有解释、分解、推导、代码、例题、数学证明、调 bug 对话，以及无数“前一句会强约束后一句”的语言轨迹。模型如果想把后续 token 预测好，就必须压缩一部分生成这些轨迹的结构。

可以打个比方。假设你只训练一个系统去预测海量国际象棋书籍、棋谱和注释中的下一个词，它也许最终会内化相当多的战术模式、开局结构和局面评估能力，尽管训练目标里从来没有显式写一个 minimax 搜索器进去。训练目标是间接的，但数据分布本身带着结构。

这能解释两个很重要的现象：

- LLM 往往在我们还没找到正确提示方式前，就已经具备某种**潜在能力（latent competence）**。
- 更好的 prompting，很多时候不是“凭空创造能力”，而是把模型内部已经部分存在的能力“叫出来”。

Wei 等人的 Chain-of-Thought 论文正是展示了这一点。让大型模型生成中间推理步骤，可以显著提升多步任务表现。真正关键的发现，不只是“模型会说更多了”，而是：在某些任务区间里，让中间轨迹显式化，能把本来藏在模型内部的计算更有效地调动出来。

但也别走到另一个极端。next-token prediction 并不会神奇地自动给我们一个稳健推理器。它提供的是一个能**涌现出某些推理行为**的基础，而不是一个干净、严格、可证明正确的推理引擎。

## 3. CoT 更像是隐藏计算的窗口，而不是“忠实推理”的证明

**一句话总结**：CoT 往往能提升表现，因为它扩展了中间计算空间，但可见的文字链条并不总是真正导致答案的内部因果过程。

Chain-of-Thought prompting 的核心，是让模型把中间步骤显式写出来。最经典的形式是：*“Let’s think step by step.”* 更可控的形式则会给几个示例，展示如何把问题分解成多个步骤。

为什么这样会有效？

### 3.1 它给了模型更多“计算空间”

如果正确答案依赖多个中间变量，那么让模型直接从题目跳到最终答案，等于要求它一次完成全部内部转换。CoT 增加了额外 token，这些 token 可以被看作临时工作区。也就是说，模型在真正输出最终答案前，有了更多机会逐步变换内部表示。

### 3.2 它把结构化问题“线性化”了

许多难题本质上可以拆成序列：先识别事实，再推出子结果，再组合，最后给答案。CoT 把隐式的约束图，展开成显式的 token 序列。对解码器来说，这通常更容易处理。

### 3.3 它让“不确定时的答案选择”更稳一些

当模型把步骤写出来后，一些错误分支会在文本层面暴露出不一致之处，这给了后续 token 更多机会去修正前面的模糊判断。尤其当它和 self-consistency 结合时，效果常常更明显。

![图 2：为什么显式推理在难题上更有帮助](images/day22/cot-vs-difficulty.png)
*图注：对简单问题，直接作答可能就够了；但随着任务难度上升，显式推理和 self-consistency 往往更有帮助。*

但这里有一个关键陷阱：可见的推理链条，并不保证就是“真实内部过程”的逐字转录。

模型可能先偏向了某个答案，再反向补写一段看起来合理的解释。它可能会合理化（rationalize），也可能会写出表面工整、实际无关的步骤。它甚至可能“答案对了，但理由不对”。这也是为什么研究里常常区分两件事：**推理轨迹是否提升了性能**，以及**推理轨迹是否忠实反映了模型内部过程（faithful）**。

> **大白话说，这其实是两个不同问题：**
> - **推理轨迹是否提升了性能** = 让模型把中间步骤写出来之后，最终正确率有没有变高？
> - **推理轨迹是否忠实** = 模型写出来的这些步骤，是否真的就是它内部得到答案的真实路径？
>
> 一个学生有时会先凭直觉猜到答案，然后再补一段看起来很工整的解题过程。这样做也许能交出一份更像样的答案，甚至最终答案也是对的，但那段“过程”并不一定忠实反映他真正是怎么想到答案的。研究者之所以要把这两件事分开，就是因为前者可能成立，而后者未必成立。

可以把它想象成学生做题。有的学生先凭直觉猜到了答案，然后再写一段漂亮的推导给老师看。那段推导可能形式正确，答案甚至也是对的，但它并不一定真的是得出答案时所用的过程。

这点对安全性和可解释性都很重要。如果我们把每一条自然语言推理链都当作“内部机制的真实记录”，就会高估系统到底有多透明。

## 4. Self-consistency、搜索，以及 test-time compute

**一句话总结**：很多推理增益，并不是来自“一条完美推理链”，而是来自多条链的采样、比较，以及在推理时额外花费的计算量。

Wang 等人的 self-consistency 思想非常优雅。它不是贪心地接受模型给出的第一条推理链，而是采样多条不同的推理路径，然后选择最终答案中“最一致”的那个。

为什么这会有效？因为在很多困难推理题里，正确答案往往能由多条不同路径收敛得到，而错误答案则更分散、更偶然。如果若干条不同的推理轨迹都导向同一个结果，这个结果通常更值得信任。

形式化一点地说，如果我们把隐藏的推理路径记作 $r$，最终答案记作 $y$，那么真正想要的不是一条贪心路径，而更像是

$$
P(y \mid x) = \sum_{r} P(y \mid r, x) P(r \mid x).
$$

self-consistency 的做法，就是通过采样多条路径并聚合答案，来近似这个边缘化过程。

这背后其实引出了一个现在非常核心的主题：**test-time compute（测试时计算）**。也就是说，不一定非要靠训练一个更大的基础模型才能提升推理能力，你也可以在推理阶段给同一个 prompt 多花一些算力，方式包括：

- 采样多条推理轨迹，
- 对候选答案做验证，
- 在搜索树上分支和剪枝，
- 调用外部工具，
- 或者执行 critique-and-revision（批判与修正）循环。

![图 3：测试时计算有用，但也有代价](images/day22/test-time-compute-tradeoff.png)
*图注：更多搜索或验证通常能提升推理准确率，但延迟和成本往往涨得更快。*

这是这个领域一个很大的观念转变。过去大家更多关注 training-time scaling，也就是更大的模型、更大的数据集。现在在推理任务上，inference-time scaling 同样关键。一个模型面对难题时，如果允许它多探索几种可能、做一点检查，它的表现往往会明显更好。这很像人类真正遇到难题时，不是一眼拍答案，而是会停一下、打草稿、排除错误路线。

当然，代价也很明显。更多 test-time compute 意味着更高延迟和更高成本。在实际系统里，你通常不希望每个请求都进入“最大推理模式”。更合理的是**自适应推理**：简单题便宜快速回答，高风险或高难度问题再启动更慎重的处理流程。

## 5. 当前 LLM 的推理到底卡在哪里

**一句话总结**：今天的模型仍然经常在精确符号状态、长程规划一致性，以及对自身中间步骤的可靠验证上掉链子。

看到推理 benchmark 一路上涨，很容易让人误以为争论已经结束。但我觉得，这种乐观还太早。

至少有五类常见失败模式反复出现。

### 5.1 算术和符号推理的脆弱性

LLM 在某些算术任务上已经相当不错，尤其当你给它 scratchpad 时更明显，但它依然很脆。精确的符号操作非常不宽容。一个进位算错、一次变量替换错了，整条解答链就可能崩掉。

这是因为神经序列模型本质上是近似函数学习器，而不是天然严格的符号机器。有时候它能把算法“模拟得足够像”，有时候就会漂。

### 5.2 过早的启发式收敛

模型可能太早抓住一个“看起来像”的题型模板。它不是认真搜索，而是先识别出表面相似性，然后快速套用一个熟悉模式。人类也会这样，但 LLM 尤其容易被“局部最 plausible 的 continuation”牵着走。

### 5.3 Faithfulness 缺口

模型写出来的步骤，不一定就是它真正得到答案的因果路径。这会让 debug 变难。因为如果那条链本身就是部分合理化的，你看它时获得的信息就没你以为的那么多。

### 5.4 长程规划能力不足

跨很多约束的多步规划，仍然是弱项。即便每个局部步骤都看起来合理，整体方案也可能违反十步前提到的条件。长轨迹下维持一致世界状态，依然很难。

### 5.5 自我验证并不总可靠

很多人会说，“那就让模型自己检查一下呀。” 有时确实有帮助，但也经常没那么好使。如果产生错误的偏差，在验证阶段仍然被同样的模型复用，那么 verifier 很可能只是用更正式的口气，再把错误确认一遍。

![图 4：很多不同问题都会被笼统地叫做“推理差”](images/day22/reasoning-failure-modes.png)
*图注：所谓“推理失败”，背后可能是算术失误、不忠实推理链、过早套模板、缺少工具，或者验证器太弱。*

一个比较稳妥的心智模型是：**今天的 LLM 推理能力很惊艳，但并不均匀。** 它已经强到足够有用，但还远没有强到可以被当作系统化、稳健的推理器去无条件信任。

## 6. 为什么工具常常比“多想一会儿”更重要

**一句话总结**：真实系统中的可靠推理，常常不是靠模型单独“想得更深”，而是靠外部工具提供精确计算、搜索、记忆和证据支撑。

假设你让模型完成以下任务：计算两个大整数的乘积、搜索代码库、比对合同条款，然后给出最终建议。我们真的希望它把所有事情都在内部完成吗？

通常不希望。

计算器做算术更可靠。检索系统做证据查找更可靠。符号求解器处理精确约束更可靠。编译器和测试套件检查代码更可靠。LLM 更擅长的地方，往往是任务编排：拆解问题、决定调用什么工具、把工具结果整合起来，并用人类易懂的语言给出解释。

这也是为什么现在越来越多系统都长成下面这个样子：

$$
\text{LLM reasoning system} = \text{base model} + \text{scratchpad} + \text{tools} + \text{verification}.
$$

这个公式当然是非正式的，但它非常符合工程现实。如果你真的关心“稳健推理”，最后几乎一定会落到混合系统上。

可以把 LLM 想成一个见多识广、总结能力很强，但在精确性上不完美的聪明实习生。你会很愿意让它起草方案、归纳证据、提出假设。但对于最后的表格总数、法律引用、或者生产环境代码补丁，你仍然会想要工具、测试和复核。

这个视角也有助于缓和那场哲学争论。如果你说的“真正推理”是“整个系统能可靠解决困难任务”，那么带工具的 LLM 系统显然已经是推理系统了。如果你说的“真正推理”是“裸基础模型在没有外部帮助的情况下，能稳定进行忠实、算法化的内部推理”，那答案就复杂得多。

## 7. 语言化推理，真的等于内部推理吗？

**一句话总结**：自然语言推理链是很有用的控制与调试接口，但它只是内部计算的近似代理，有时甚至会误导你。

这一节是争议真正微妙的地方。

当模型写出“第一步先算总成本，第二步再减去折扣”时，我们很容易脑补一个整洁的小型符号机器在网络内部按这个顺序运行。现实有时确实会部分对齐，但也完全可能不对齐。

为什么会不对齐？

1. **语言本身是有损接口**。内部激活是高维向量，不是英文句子。
2. **模型可以事后合理化**。一旦它已经偏向某个答案，就可能再生成一段看起来很像“原因”的解释。
3. **不同内部路径可能映射成相似的文字链条**。也就是说，外部看起来一样，内部其实不一定一样。

那是不是因为 CoT 不是百分百 faithful，我们就该忽略它？也不是。那会走到另一个极端。

CoT 至少仍然有四个很实用的价值：

- 它经常能提升准确率；
- 它让某些错误更容易被人类发现；
- 它提供了一个可控的分解接口；
- 它还能把中间过程喂给下游 verifier 或工具。

更合适的态度是务实一点。把 reasoning trace 看成一个聪明但并非完全可靠组件写出来的 debug log。它有价值，但不神圣。

## 8. 常见误解

**一句话总结**：最大的误解，通常来自把 LLM 推理能力看成“已经完全解决”或者“完全是假的”这两个极端。

### 误解 1：“既然只是 next-token prediction，就不可能推理。”

这个说法太绝对了。简单目标在大规模数据和足够容量下，可以涌现出非常复杂的行为。数据里的结构，会把模型逼向比“纯表面模仿”更深的能力。

### 误解 2：“只要它写出了推理链，就证明它真的推理了。”

也太绝对了。可读的推理链也许有帮助，但它不是“忠实内部机制”的充分证据。

### 误解 3：“所谓推理，其实全是记忆。”

记忆当然重要，但它解释不了全部现象。尤其是在一些新的组合任务上，模型配合搜索或工具仍然能泛化，这说明不只是最近邻检索那么简单。

### 误解 4：“模型再大一点，推理问题自然就都解决了。”

规模确实帮助很大，但脆弱的精确性和验证问题仍然存在。单纯 scaling，替代不了好的系统设计。

### 误解 5：“benchmark 上推理强，线上就一定可靠。”

Benchmark 往往是干净、狭窄、定义清楚的任务。真实系统面对的是噪声输入、模糊目标、缺失证据、工具错误、延迟约束，以及不断变化的环境。

---

## 9. 对构建者的实际启示

**一句话总结**：如果你在用 LLM 搭系统，最好把推理能力视为“很有价值，但不均匀”，并围绕它做支持、验证和约束。

下面这些经验，是我觉得在工程里最值得信的。

1. **简单任务直接答，复杂任务进入更谨慎模式。** 推理不该永远开到最大，而应该按需启用。
2. **精确性重要时，优先把工作交给工具。** 算术、检索、代码执行、形式化检查，不该全靠模型脑补。
3. **尽量让中间结果结构化。** 表格、计划、方程状态、带引用的证据，比自由散文更容易验证。
4. **不要把“解释写得好”误当成“答案更对”。** 漂亮的 reasoning trace 仍然可能指向错误结论。
5. **认真建设 verifier。** 对很多应用来说，一个强 verifier 的价值，不比一个强 generator 小。
6. **评估完整系统，而不是只看基础模型。** 真正重要的问题是，在真实约束下，你的整个 pipeline 能不能稳定到达正确结果。

一个很实用的经验法则是：如果错误代价很高，就把模型的第一版答案当成 draft（草稿），不要把它当 verdict（终审结论）。

## 10. 收束思考

LLM 让传统意义上“模式匹配”和“推理”的边界，变得比很多人预想的模糊得多。它们显然已经不只是浅层 autocomplete 了。但与此同时，它们也还远不是那种可以脱离支架、完全值得信任的系统化符号推理器。

如果要给出我的判断，我会说：现代 LLM 具备的是一种**部分的、概率式的、可被工具放大的推理能力**。这个答案没有两极观点那么戏剧化，但我觉得更诚实。它们确实能在很多情境下做出惊人的分解、搜索和推断；可一旦涉及精确状态跟踪、长程一致性和忠实自解释，脆弱性仍然很明显。

真正令人兴奋的地方在于，这不只是“模型本身”的问题，也是一道“系统设计”的问题。更好的 prompting、self-consistency、verifier、retrieval 和工具集成，都能明显抬高能力上限。换句话说，未来的推理能力，也许不只是由 base model 决定，还很大程度取决于**我们如何把模型包进一个更好的系统里**。

这也自然连接到课程后面的主题。一旦你开始在意“如何在不确定条件下推理”，你很快就会遇到 evaluation、tool use、world models，最后走向 agents。一个好的 agent，从来不只是“一个会说话的模型”，而是一个能规划、行动、观察并修正的循环系统。

所以，LLM 真的会推理吗？

一个谨慎而诚实的回答是：**有时会，不完美地会，而且通常在“给它一点思考空间，再给它一点外部校验”的时候会得最好。**
